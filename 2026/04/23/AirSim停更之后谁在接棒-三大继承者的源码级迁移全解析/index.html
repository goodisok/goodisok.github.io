AirSim 停更之后，谁在接棒：三大继承者的源码级迁移全解析

发表于2026-04-23|更新于2026-04-23|仿真开源项目
|浏览量:

2022 年 7 月，微软宣布停止维护 AirSim，18K Star 的仿真平台一夜之间变成”弃子”。四年后的今天（2026 年 4 月），AirSim 生态分裂为三条路线：社区 fork 的 Colosseum 在 AirSim 代码基础上逐版本追赶到 UE5.6；前微软 AirSim 团队组建的 IAMAI Simulations 将微软内部的 Project AirSim 以 MIT 协议开源，架构完全重写，支持 UE5.7；比利时 Cosys-Lab 的 Cosys-AirSim 则在传感器维度大幅扩展，新增 GPU LiDAR、Echo 声纳、实例分割等研究级功能，支持 UE5.5。

本文从时间线、架构变更、源码级 diff、已知问题四个维度，完整记录这次社区接力的技术细节。如果你正在纠结”AirSim 还能不能用””该选哪个继承者”，这篇文章会给你一个清晰的答案。

1.3 Project AirSim 的重生

2023 年底微软裁撤 Project AirSim 团队后，原团队的核心工程师并没有放弃，而是组建了 IAMAI Simulations（IAMAI CONSULTING CORP），于 2025 年 5 月将 Project AirSim 以 MIT 协议重新开源。值得注意的是，2026 年 3 月微软在原始 AirSim 仓库中更新了 README，正式认可了 IAMAI 的工作：

“Microsoft and IAMAI collaborated to advance high-fidelity autonomy simulations through Project AirSim — the evolution of AirSim — released under the MIT license as part of a DARPA-supported initiative. IAMAI is proud to have contributed to these efforts and has published its version of the Project AirSim repository at github.com/iamaisim/ProjectAirSim.”
— microsoft/AirSim README（2026-03-15 更新）

这段话透露了三个重要信息：

  1. 微软官方背书：Project AirSim 不是前员工的”野路子项目”，而是获得微软认可的正统继承
  2. DARPA 资助：该项目是美国国防部高级研究计划局（DARPA）资助项目的一部分，意味着有军事级自主系统仿真的背景
  3. MIT 开源：明确了许可证，消除了潜在的法律顾虑

二、核心架构变更：Colosseum 到底改了什么

2.1 引擎版本跃迁：UE4.27 → UE5.x

这是 Colosseum 最大的工程量。UE4 到 UE5 不是简单的版本号升级——Epic 在 UE5 中重写了大量子系统，Colosseum 必须逐一适配。

2.1.1 RHI（渲染硬件接口）变更

UE5 重命名了一批 RHI Buffer 类型。Colosseum v2.1.0 的核心改动：

UE5 将 FRHITexture2D 统一为 FRHITexture，具体类型通过 GetDesc().Dimension 判断。所有直接使用 FRHITexture2D* 的地方都需要改为 FTextureRHIRef 或 FRHITexture*。

2.1.2 CameraDirector 重命名

UE5.3 引擎内部新增了一个同名的 CameraDirector 类，与 AirSim 的 CameraDirector 冲突。Colosseum v2.2.0 的解决方案：

所有引用该类的蓝图、C++ 文件都需要同步更新。

2.1.3 构建系统适配

UE5 的 Build.cs / Target.cs 使用了新的版本枚举：

这里有一个常见的社区踩坑点：如果硬编码为 BuildSettingsVersion.V5 或 EngineIncludeOrderVersion.Unreal5_4，在非 5.4 的引擎上会直接编译失败。Colosseum 的部分提交初期犯了这个错误，后续 Issue #98 中社区反馈后改为 Latest。

2.2 物理引擎迁移：PhysX → Chaos

这是 AirSim → Colosseum 最深层的架构变更。

2.2.1 背景

  • AirSim：UE4 默认使用 NVIDIA PhysX 作为物理引擎
  • UE5：Epic 自研 Chaos Physics 替代 PhysX，UE5.1 起 PhysX 被标记为 deprecated

2.2.2 车辆物理的迁移

Colosseum v2.0.0-beta.0（2022-11）最早的改动就是车辆资产的 Chaos 迁移：

组件 AirSim (PhysX) Colosseum (Chaos)
车辆蓝图 WheeledVehicle (PhysX Vehicle) ChaosWheeledVehicle
车轮 UVehicleWheel UChaosVehicleWheel
移动组件 UWheeledVehicleMovementComponent UChaosWheeledVehicleMovementComponent
物理资产 PhysX PhysicsAsset Chaos PhysicsAsset
SUV 模型 PhysX 版本 重新导入并配置 Chaos 碰撞体

v2.0.0 Release Note 明确写道：

Convert SUV to Chaos Physics from PhysX. Vehicle now uses Chaos Physics and related assets.

2.2.3 对无人机的影响

关键事实：AirSim 的无人机物理不走 UE 物理引擎。AirSim 有自己的 FastPhysicsEngine，是一个纯 C++ 头文件库，独立于 UE 的 PhysX/Chaos 之外运行。因此：

  • 无人机的飞行动力学、螺旋桨模型、气动力计算 → 不受 PhysX → Chaos 迁移影响
  • 无人机的碰撞检测 → 调用 UE 的 LineTrace / SweepSingleByChannel → 需要适配 Chaos，但 UE 层面做了兼容
  • 车辆仿真 → 完全受影响，需要全部重写车辆蓝图

三、源码级 diff 详解

下面按文件/模块级别梳理 Colosseum 相对于 AirSim v1.8.1 的关键改动。

3.1 Unreal 插件层 (Unreal/Plugins/AirSim/)

3.1.1 相机系统

文件：Source/AirSim/Private/SimMode/SimModeBase.cpp，PIPCamera.cpp，CameraDirector.cpp

改动 原因
ACameraDirector → AAirSimCameraDirector 避免 UE5.3+ 命名冲突
PIPCamera::EndPlay 中添加 Super::EndPlay() 调用 修复 UE5.6 中退出时崩溃
CameraManager 更新 EndPlay 逻辑 防止 EndPlay 阶段访问已销毁对象

PIPCamera 的修复是一个典型的 UE5 迁移问题。UE4 中 EndPlay 不调用父类通常不会出问题（只是不规范），但 UE5 加强了生命周期管理，不调用 Super::EndPlay() 会导致组件清理不完整，引发 nullptr 崩溃。

3.1.2 RHI 与渲染

文件：Source/AirSim/Private/Sensors/SensorBase.cpp 及相关渲染文件

3.1.3 BlocksV2 环境

AirSim 原版使用 Blocks 环境，Colosseum 在 2024 年 7 月将其替换为 BlocksV2：

  • 更新 .uproject 文件的 EngineAssociation 字段
  • 更新 Target.cs / Editor.Target.cs 使用新的 BuildSettingsVersion
  • 移除对 ModelingToolsEditorMode 插件的依赖（某些 UE 版本中该插件不可用）

3.2 AirLib 核心库 (AirLib/)

好消息：AirLib 几乎没有改动。

AirSim 的架构设计在这里体现了远见——AirLib 是纯 C++ 头文件库，不依赖 UE API。因此 UE4 → UE5 的迁移对它基本无影响。Colosseum 对 AirLib 的修改仅限于：

文件 改动
include/vehicles/multirotor/api/MultirotorApiBase.hpp 修复 getMultirotorState 返回值类型不匹配
其他 极少量，主要是编译器 warning 修复

这意味着：如果你之前基于 AirLib 做了自定义物理模型或传感器扩展，迁移到 Colosseum 时这部分代码几乎不需要改动。

3.3 Python 客户端 (PythonClient/)

文件：airsim/types.py

Colosseum 在 2024 年集中修复了 types.py 中的多个 Bug，这些 Bug 在 AirSim 的 Issue 列表中存在多年但从未被修复。

3.4 构建脚本

文件：build.cmd (Windows)，setup.sh / build.sh (Linux)

改动 说明
移除 popd 路径问题 Windows 路径包含空格时 build 脚本出错
更新 Clang 版本检测 Linux 上编译时需匹配 UE5 的 Clang 版本
添加 Ubuntu 24.04 支持 新增 apt 依赖列表
更新 VS 版本检测 从 VS 2019 改为 VS 2022

3.5 新增 API

API 版本 说明
listActors() v2.2.0 列出场景中所有 Actor，方便调试和场景管理

新增 API 不多，Colosseum 的重心在”让现有功能在 UE5 上正常工作”而非添加新功能。

四、被砍掉的功能

迁移到 UE5 的过程中，有些功能被暂时或永久移除：

功能 状态 原因
OpticalFlow（光流相机） ❌ 移除 UE5 中 motion vector 的变换矩阵发生变化，光流输出不正确。v2.0.0 Release Note：”Remove support for OpticalFlow until resolution found with various transforms”
macOS 支持 ⚠️ 实验性 Apple 的构建工具链频繁变化，M1/M2 芯片兼容问题。README 声明：”There are ongoing discussions to remove this support”
预编译二进制环境 ⚠️ 仅 Blocks (UE5.1) AirSim 曾提供多个预编译环境（Neighborhood、Africa 等），Colosseum 只提供了一个 Blocks 的 Win64 可执行文件

五、社区 fork 生态：当 Colosseum 也不够用时

Colosseum 的维护节奏较慢（2025 年全年只有约 10 个 commit），社区中出现了多个针对特定需求的 fork。

5.1 典型的社区 fork 定制

以笔者自己的 fork goodisok/Colosseum 为例，分支 feature/jpeg-geomag-px4 在 Colosseum 基础上增加了：

  • 可调质量 JPEG 压缩传输：将 1080p 图像传输从 ~10 FPS 提升到 ~22 FPS
  • 地磁修正：解决自定义 OriginGeopoint 后 PX4 无法解锁的问题
  • UE5.7 编译适配：在官方 UE5.6 支持的基础上向前推进一个版本

这些改动都是笔者在实际截击机仿真项目中遇到的刚需问题，上游 Colosseum 短期内不太可能合并。

5.2 UE5.7 编译适配（Issue #144）

截至 2026 年 4 月，Colosseum 主分支只支持到 UE5.6。社区已提交 PR #144 修复 UE5.7 的编译问题，核心改动：

  • 修复 UE5.6+ 废弃的 API 调用
  • 编译 AirSim 为 UE5.7.1 的引擎插件已通过验证

但这个 PR 尚未合并。如果你需要使用 UE5.7，目前的选项是：

  1. 手动 cherry-pick PR #144 的 commit 0598cb9
  2. 使用笔者的 fork goodisok/Colosseum，分支 feature/jpeg-geomag-px4 已包含 UE5.7 支持
  3. 等待上游合并（时间不确定）

七、Colosseum 的现状与未来

7.1 坦率的评估

Colosseum 的优点：

  • 成功完成了 UE4 → UE5 的迁移
  • AirLib 核心物理引擎未被修改，保持了与 AirSim 的 API 兼容性
  • MIT 许可证，完全开源
  • 迁移成本低——原有 AirSim 代码几乎不需要改动

Colosseum 的问题：

  • 维护力度有限：2025 年全年约 10 个 commit，核心贡献者仅 2–3 人
  • 2025 年 12 月 Codex Labs 退出维护者行列，项目未来走向不明
  • 文档严重滞后，很多页面仍是 AirSim 时代的内容
  • 缺少 CI 测试覆盖，部分 Build 状态已 broken
  • 新功能开发几乎停滞，主要精力用于追赶 UE 版本更新

Cosys-AirSim 的优点：

  • 功能扩展最丰富：GPU LiDAR、Echo 声纳、实例分割、多层标注
  • 研究级传感器仿真——适合生成高质量训练数据
  • 内置地面机器人模型（Husky、P3DX）
  • 升级到 C++20，依赖库版本最新
  • 有独立文档站和商业网站（cosys-airsim.com）
  • 10 个正式 release，版本管理规范

Cosys-AirSim 的问题：

  • 学术驱动，文档声明”provided as is, will not be actively updated”
  • Python 包名改为 cosysairsim，迁移需要修改所有 import
  • 主分支 UE5.5（非最新的 UE5.7）
  • 社区规模相对较小（350 stars）

Project AirSim 的优点：

  • 由 AirSim 原始团队工程师维护，对代码最熟悉
  • 微软官方认可 + DARPA 资助背景——三个继承者中唯一获得原始作者和官方机构背书的项目
  • 架构完全重写，三层解耦更现代化
  • 活跃度高（2026 Q1 仍有持续提交）
  • 已支持 UE5.7，提供完整的 AirSim 迁移文档
  • 支持 JSBSim、MATLAB/Simulink 等更多集成

Project AirSim 的问题：

  • 仍处于早期阶段（v0.1.1），API 可能不稳定
  • 与 AirSim 的 API 不兼容——迁移需要改动代码
  • 社区生态尚未建立，第三方教程和工具极少
  • 配置格式从 settings.json 变为 JSONC，学习成本增加

7.2 时间线上的 commit 活跃度

平台 优势 劣势

八、总结

8.1 Colosseum 做了什么

Colosseum 在三年间完成了一项不小的工程：

  1. UE4 → UE5 全面迁移（UE5.0 → 5.1 → 5.2 → 5.3 → 5.4 → 5.6）
  2. PhysX → Chaos 物理引擎切换（车辆部分）
  3. 编译工具链现代化（VS 2022 / Clang 18 / C++17）
  4. 修复了 AirSim 长期存在的多个 Bug（types.py、getMultirotorState、EndPlay 崩溃等）
  5. 操作系统支持扩展（Ubuntu 24.04、Windows 11）

8.2 Colosseum 没做什么

  1. 没有新增重要功能——核心 API 与 AirSim v1.8.1 基本一致
  2. 没有修复光流相机——OpticalFlow 仍不可用
  3. 没有更新文档——大部分文档仍是 AirSim 时代的内容
  4. 没有建立完善的 CI/CD——Build 状态不稳定

8.3 一句话定性

Colosseum 是”续命版”，Cosys-AirSim 是”强化版”，Project AirSim 是”进化版”。 三者分别解决不同的问题：Colosseum 让旧项目能在 UE5 上跑；Cosys-AirSim 为研究者提供了 AirSim 从未有过的传感器能力；Project AirSim 则面向未来做了架构重写。选择时应基于你的具体需求，而非简单地”选最新的”。
