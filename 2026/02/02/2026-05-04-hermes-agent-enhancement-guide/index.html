Hermes Agent 能力增强实战：从 Skills 系统到 Karpathy 编码原则的全面升级

发表于2026-02-02|更新于2026-02-02|人工智能
|浏览量:

本文目标：本文不是一篇教程，而是一份实操增强报告——记录如何将本地的 Hermes Agent 从 v0.10.0 升级到最新版，导入 42+ 个来自社区的生产级外部 Skills，并将 Andrej Karpathy 的编码原则融入 Agent 的默认人格，从而让 AI Agent 的能力实现质的飞跃。所有操作均经过实际执行验证。

一、为什么需要增强 Hermes？

2026 年 4 月，NousResearch 的 Hermes Agent 在 GitHub Trending 上以 月增 108,122 星 的速度霸榜第二。与此同时，社区涌现了大量 Agent Skills 生态项目——mattpocock/skills（57K 星）、addyosmani/agent-skills（27K 星）、forrestchang/andrej-karpathy-skills（109K 星）——它们积累的工程经验和编码最佳实践，本质上是可以跨 Agent 复用的。

但现实是：

  • 大多数人在跑 hermes 后，用的是出厂默认的 Agent 行为——没有外部经验注入
  • Skills 系统虽支持导入，但文档分散，缺少一个端到端的实操路径
  • Karpathy 的编码原则虽在 X 上爆火，但没有直接转化为 Agent 可消费的格式

本文就是填补这个空白。

二、增强前的基线状态

在开始之前，先明确起点：

维度 增强前值
Hermes 版本 v0.10.0（落后 2278 个 commit）
Skills 数量 ~30 个（built-in）
外部 Skills 0
默认人格 无特定（出厂配置）
编码原则注入 无
GitHub 连接 HTTPS（中国网络频繁超时）

这是一个典型的”能用但不好用”的状态——Agent 能做基本的事，但它的行为完全由原始 prompt 决定，没有任何社区经验或最佳实践的积累。

四、第二步：导入外部 Skills 生态

4.1 什么是 Hermes Skills？

Hermes 的 Skills 系统本质上是一种程序性记忆——它不是数据（像 RAG 一样存文档），而是做事的方法。一个 Skill 是一个 SKILL.md 文件，定义了：

  • 何时触发（description 字段）
  • 怎么做（步骤、命令、脚本）
  • 注意事项（pitfalls）

Skills 系统的核心价值在于：将一次成功的复杂操作固化为可重复调用的 Procedure，从此不再需要从零推理。

4.2 选择要导入的外部 Skills

五、第三步：构建 Karpathy 编码原则 Skill

5.1 四大原则

Andrej Karpathy（前 Tesla AI 总监、OpenAI 创始成员）在 2026 年 4 月发布了一系列关于 LLM 编码陷阱的观察，迅速在开发者社区引发轰动。他将 AI Agent 编码中最常见的问题归纳为四个核心原则：

原则 1 — 先思考再编码（Think Before Coding）

大多数 LLM 犯的错误不是因为不会写代码，而是没想清楚就写。

  • 明确陈述假设——如果不确定，先问清楚
  • 呈现多个解读——不要默默选择一种
  • 有更简单的方案时主动提出
  • 困惑时停下来——说出不清楚的地方再问

原则 2 — 简洁优先（Simplicity First）

最小代码原则。每一行都要有存在的理由。

  • 只写解决当前问题所需的最少代码
  • 不为只用一次的场景做抽象
  • 不添加未要求的”灵活性”或”可配置性”
  • 不处理不可能发生的错误场景
  • 如果 200 行能写成 50 行，就重写

原则 3 — 外科手术式改动（Surgical Changes）

优秀的工程师像外科医生——只切该切的地方。

  • 只动必须动的地方
  • 不”顺带改进”不相关的代码、注释或格式
  • 不重构没坏的东西
  • 匹配现有风格
  • 只删除自己改动产生的孤儿代码

原则 4 — 目标驱动执行（Goal-Driven Execution）

把”做任务”变成”达到目标”。

  • 把任务转化为可验证的目标：”修 bug” → “写一个能复现 bug 的测试并让它通过”
  • 多步骤任务格式：1. [步骤] → 验证: [检查项]
  • 强验证标准让 Agent 能独立迭代

六、核心机制解析

6.1 Skills 是如何被调用的？

理解了 Skills 的运作机制，才能最大化其价值。

加载阶段： 对话开始时，Hermes 扫描 ~/.hermes/skills/ 下所有 SKILL.md 文件，提取 name 和 description 字段，放在 Agent 的系统提示中作为可用的技能列表。

调用阶段： 当用户提出请求时，Agent 会根据技能描述判断哪个 Skill 适用，然后用 skill_view(name) 工具加载对应 SKILL.md 的完整内容。

这就意味着：

  • 描述一定要写准——太模糊导致 Agent 不知道该不该用
  • 内容要精炼——不必要的废话会占用宝贵的上下文 token
  • Skill 可以嵌套——一个 Skill 的执行过程中可以调用另一个 Skill

6.2 Personality 与 Skills 的协同

Personality 和 Skills 虽然都影响 Agent 行为，但角色不同：

维度 Personality Skill
作用域 全局（整个对话） 任务级（按需加载）
加载时机 对话开始（注入 system prompt） 运行时（按需调用）
内容类型 行为风格、价值观、原则 具体步骤、命令、流程
冲突处理 不可叠加（一次只能选一个人格） 可叠加（多个 Skill 同时生效）

我们当前的配置是：

  • Personality 层：Karpathy 四大原则（约束编码行为）
  • Skills 库：42+ 个外部 + 30+ builtin = 70+ 个可用技能（解决具体问题）

当 Agent 收到”帮我做个代码审查”的任务时：

  1. Karpathy 人格约束它先思考再动笔（原则 1）
  2. 它自动匹配到 code-review-and-quality skill
  3. 按 Skill 定义的步骤执行审查

这种协同效应远超单独使用其中任何一个。

7.2 定性变化

增强前的 Hermes 行为特征：

  • 接到编码任务后直接开始写
  • 经常做过多的抽象和重构
  • 代码风格与项目不匹配
  • 完成任务后缺少验证

增强后的行为特征：

  • 先问清楚假设再动手（Karpathy 原则 1）
  • 只写最精简的代码，不引入不必要的设计模式（原则 2）
  • 只改动相关代码，不顺手修饰（原则 3）
  • 每一步都有验证标准（原则 4）
  • 遇到复杂任务自动加载对应 Skill 按步骤执行

7.3 典型场景

场景：质量检查 2000+ 行代码 → 先用 code-review-and-quality + security-and-hardening + performance-optimization 三个 skill 并行审查，然后 Karpathy 原则约束改动量（只针对必改项，不顺手重构）。

8.2 Skills 兼容性

外部 Skills 是为 Claude Code 设计的（.claude/, CLAUDE.md 格式），Hermes 的 Skills 系统使用 SKILL.md 格式。两者结构相似但不完全一致：

格式维度 Claude Code 格式 Hermes 格式
文件名 CLAUDE.md 或 SKILL.md SKILL.md
目录结构 .claude/skills/ ~/.hermes/skills/<name>/
元数据 --- frontmatter --- YAML frontmatter
触发机制 文件名匹配 description 字段语义匹配

关键发现： Hermes 的 Skills 系统实际上兼容 Claude Code 的 skill 目录结构。只要源仓库使用 skills/<name>/SKILL.md 的组织方式，即可直接克隆到 ~/.hermes/skills/ 下被 Hermes 识别。这解释了为什么 mattpocock/skills 和 addyosmani/agent-skills 的 skills/ 目录都能无缝导入。

九、Skills 系统的设计哲学

9.1 为什么是 Skills 而不是 RAG？

这是一个架构选择问题。很多 Agent 框架依赖 RAG（检索增强生成）来”记住”如何做事，但 Skills 走的是另一条路：

维度 RAG Skills
存储内容 文档、知识、事实 程序、步骤、流程
检索方式 向量相似度搜索 语义描述匹配
输出 参考信息 可执行的指令序列
上下文消耗 全文注入（高） 按需加载（低）
维护方式 重新索引 编辑 SKILL.md
适用场景 “告诉我 X 是什么” “帮我做 X 这件事”

Skills 的核心优势在于：它不提供”知识”，它提供”方法”。对于 AI Agent 来说，”知道怎么干”比”知道是什么”重要得多。

9.2 Skills 的自我进化

Hermes Skills 系统有一个独特的设计：Agent 可以自己创建和更新 Skills。

  • 解复杂的任务后，Agent 可以调用 skill_manage(action='create') 保存工作流
  • 发现 Skill 过时或不完整时，可以直接 skill_manage(action='patch') 修复
  • 外部 Skills 通过 Tap 机制同步社区最新版本

这意味着 Skills 库不是静态的——它在使用过程中持续增长。每次成功的复杂操作都是技能的固化，每次遇错都是技能的修正。这本质上是一个技能积累飞轮。

十、未来演进方向

10.1 Kanban + Skills 的工作流编排

Hermes v0.12.0 新增的 Kanban 系统可以与 Skills 深度结合。理想的工作流是：

  1. Goal（大目标） → 拆解为 Kanban 卡
  2. 每张卡自动匹配合适的 Skill
  3. Skill 指导 Agent 完成任务
  4. 完成后卡片自动流转

10.2 外部 Skills 的版本管理

当前 Tap 机制没有版本锁定。当源仓库大改时，可能导致 Skill 不兼容。未来需要：

  • 支持指定 Tap 的 commit/tag
  • Skills 更新前自动 diff 展示变化
  • 版本回退能力

10.3 Agent 间的 Skills 共享

当前 Skills 是本地私有的。如果能在不同 Agent 实例之间共享 Skills：

  • 团队沉淀的工程经验直接复用到所有 Agent
  • 更快的技能进化速度
  • 将 Skills 变成团队的工程资产

十一、总结

本文记录了将 Hermes Agent 从 v0.10.0 升级到 v0.12.0、导入 42+ 个社区 Skills、并将 Karpathy 编码原则注入默认人格的完整过程。

核心贡献不在于单个操作——每个步骤本身都有官方文档——而在于：

  1. 打通了社区 Skills 到 Hermes 的导入链路，验证了跨 Agent 框架的 Skill 兼容性
  2. 构建了”Personality 约束行为 + Skills 指导步骤”的协同架构
  3. 将 Karpathy 的编码原则从推文转化为 Agent 可消费的形式

最终的 Hermes 不再是出厂配置的空壳——它带着社区积累的工程经验、被编码原则约束的行为规范，以及持续进化的能力。这就是”自我增强的 Agent”的起点。
